# DH 如何工作

DH 不阅读一段自然语言后猜测“它是不是正确”。它只检查项目能够显式表达、重复验证的关系。

## 输入

每次运行从指定项目根目录读取：

- `docs-hygiene.yml` 中的治理策略；
- 受管 Markdown、frontmatter 和 Manifest；
- 稳定身份及其 canonical/localized 表示；
- 语义引用、Pin、生命周期和所有权声明；
- 由 Adapter 提供的外部工具结果。

项目目录是运行边界。Git 仓库只是物理载体；DH 不会自动遍历 monorepo 中的所有项目。

## 求值过程

```text
项目事实
   ↓
规则是否适用
   ↓
自动激活建议 + 显式项目策略
   ↓
确定性检查器
   ↓
finding / 画像 / 影响集合
```

同一份输入和策略应当产生相同结果。需要概率判断的工作可以消费 DH 输出，但不会被隐藏在阻断性规则内部。

## 三种变化结果

依赖分析区分三件容易混淆的事：

```text
potentialImpact = 图上可以到达的下游消费者
invalidated     = 锁定的上游状态已经变化
reviewSet       = 策略要求处理的失效消费者
```

普通引用可以告诉团队“谁可能受影响”；Pin 和内容锚提供“所审阅状态已经变化”的确定性证据；项目策略再决定哪些失效必须进入复核。

## finding 不是业务结论

DH 可以确定：

- 一个稳定身份没有对应权威来源；
- 一个语义引用无法解析到预期类型；
- 已 Pin 的内容与所审阅状态不一致；
- localized 表示缺少对应 canonical 身份；
- 一个终止状态资产仍被当前文档消费。

DH 不能确定：

- 产品需求在商业上是否合理；
- 两段措辞不同的自然语言是否语义等价；
- 应该采用哪一种架构方案；
- 团队应该接受还是拒绝一项变更。

机器负责稳定地暴露断裂，人对含义和选择负责。
