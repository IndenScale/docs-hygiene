# “文档的 Dependabot”究竟是什么意思

“文档的 Dependabot”不是说 DH 会自动重写过期文档。这个类比指向的是另一件事：让原本不可见的知识依赖进入持续反馈回路。

## 软件依赖为什么容易治理

软件包通常具有：

- 稳定名称；
- 可识别版本；
- 显式依赖声明；
- 更新和兼容性信号；
- 自动化检查入口。

当一个依赖变化时，团队至少知道“谁依赖它”和“当前锁定状态是否过期”。至于是否升级、怎样迁移，仍然由人或更高层自动化决定。

## 知识依赖缺少同样的反馈

项目文档经常通过自然语言和文件路径形成隐含关系：需求引用一个术语，设计依赖一个约束，Issue 声称交付某项需求，但这些关系没有稳定身份，也没有新鲜度证据。

上游发生变化后，下游内容仍然存在，CI 也保持绿色。偏差只有在事故、交接或下一轮开发时才被重新发现。

## DH 对应的职责

DH 把能够确定性表达的部分建立为工程机制：

| 软件依赖治理 | 文档依赖治理 |
| --- | --- |
| Package identity | 稳定文档或概念身份 |
| Dependency declaration | Reference / Dependency |
| Lockfile state | Pin / 内容锚 |
| Outdated signal | invalidated finding |
| Dependency graph | 文档治理关系图 |
| Update review | 策略选择的复核集合 |

这个映射不是要把文档伪装成代码，而是把“谁依赖谁、所审阅状态是否变化”从人的记忆变成可检查事实。

## 为什么 DH 不自动修复含义

软件包版本可以被机器比较，业务含义却不能仅靠字符串差异决定。上游定义变化后，下游可能需要修改，也可能经过复核后确认不受影响。

DH 的责任是在正确的位置产生明确 finding，给出上游、消费者、关系、状态和影响范围。它不应把概率性的自然语言判断隐藏成确定性结论。

外部 Agent 可以读取 finding、分析正文并提出修改，但最终证据仍需要回到显式关系和人的判断。

## 与回溯工程的关系

回溯工程从历史代码、Issue 和事故中恢复系统为何变成今天这样，并重建正向设计。DH 接管的是后续持续治理：保持这些新形成的知识资产有身份、依赖、新鲜度和演化关系。

前者解决“我们怎样恢复正确的设计”，后者解决“怎样不让它再次静默过期”。

所以“文档的 Dependabot”是一个入口，而不是完整定义。DH 最终检查的是项目文档之间是否仍然保持可信的工程关系。
