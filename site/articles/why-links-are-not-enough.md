# 为什么死链检查远远不够

一个链接返回 200，只能说明目标还在。它不能说明目标仍然是正确的定义，也不能说明引用者已经处理了上游变化。

这正是很多文档治理系统的盲区：它们能发现文件消失，却看不见知识关系失效。

## 可达不等于正确

假设网关 PRD 链接到“限流策略”文档。后来团队把旧策略归档，又在新目录创建了同名文件。为了保持链接可用，维护者更新了路径。

死链检查会通过。但我们仍然不知道：

- 新文件是不是原来那个语义对象；
- 它是否具有权威定义；
- PRD 期待的是策略、设计还是操作指南；
- 新策略变化后，PRD 是否完成复核。

路径只回答位置，不能回答身份和关系。

## 三个不同层次

文档引用至少有三个层次：

| 层次 | 回答的问题 |
| --- | --- |
| 路径导航 | 目标能否打开？ |
| 语义引用 | 当前内容消费的是哪个稳定身份？ |
| 状态锁定 | 消费者审阅的是上游哪个确定状态？ |

普通 Markdown Link 很适合第一层，不应该被强行解释成所有业务关系。需要语义治理时，项目应显式声明稳定身份和 Reference；需要精确新鲜度时，再对关键依赖增加 Pin。

## 上游变化之后

当权威定义发生变化，系统需要区分：

1. 哪些消费者在依赖图上可能受到影响；
2. 哪些消费者锁定的上游状态已经确定失效；
3. 哪些失效消费者必须由策略要求进入复核。

如果把三者混成“所有链接都需要更新”，团队会迅速被虚警淹没。如果只检查死链，真正的失效又会静默通过。

## DH 的选择

DH 保留导航与语义的区别：Markdown Link 负责路径完整性，显式语义边负责身份和依赖，Pin 负责确定性的新鲜度证据。

它不会判断新旧两段文字是否表达同一个业务含义。它能确定的是：一个消费者原来审阅过的上游状态已经变化，应该由人重新确认。

文档治理的起点不是让所有链接都能点开，而是让重要关系的失效变得可见。
