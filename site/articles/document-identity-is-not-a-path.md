# 文件路径不是文档身份

工程团队经常把路径当成文档身份：`docs/api/rate-limit.md` 就是“限流策略”。这个约定简单，直到文件开始移动、改名、拆分或产生另一种语言表示。

路径描述存储位置，身份描述语义对象。两者在小项目里经常重合，但不应该被永久绑定。

## 一次普通的目录整理

维护者把文件从：

```text
docs/api/rate-limit.md
```

移动到：

```text
docs/platform/traffic/rate-limiting.md
```

如果路径就是身份，这次整理相当于删除一个对象，再创建另一个对象。旧引用、审阅记录、生命周期和本地化关系都需要靠额外约定恢复。

如果文档拥有稳定身份 `UL-RATE-LIMIT`，路径变化只是这个身份的表示位置发生迁移。

## 稳定身份带来的能力

将身份与路径分开后，团队可以确定性地检查：

- 新旧文件是否意外声明了同一个身份；
- localized 文件是否对应同一个 canonical 身份；
- 重命名是否保留 alias 和迁移策略；
- 已终止身份是否仍被当前内容引用；
- 替代资产是否显式接管权威关系。

这些能力并不要求把 Markdown 变成数据库。文件仍然可以正常阅读、移动和版本管理，只是关键对象不再由路径偶然决定“它是谁”。

## 不要给所有文件编号

稳定身份有成本，不应机械地覆盖所有文档。临时笔记、生成结果和普通导航页未必需要长期身份。

最需要稳定身份的是会被其他资产依赖、需要跨语言表示、拥有生命周期或承担权威定义的内容。例如共享术语、长期需求和架构决策。

## Git 也不是语义身份

Git commit 很适合证明某一时刻的内容状态，却不适合作为业务对象的名字。稳定 ID 表示“是谁”，Git、hash 和 snapshot 表示“我看过哪个状态”。

分开这两个问题，文档变化才既能保持历史，也能维持当前权威。
